## 二、RAG 向量库设计详解

### 2.1 整体架构

```
文档发布
    │
    ▼
EventEmitter('doc.created' / 'doc.updated' / 'doc.deleted')
    │
    ▼
RagService（监听事件）
    │
    ├─── 删除旧向量（removeDoc）
    │       └─ 查 rag_vector_mapping → 取 vectorId 列表
    │          → ChromaDB.delete(ids)
    │          → 删除 rag_vector_mapping 行
    │
    └─── 重新入库（indexDoc）
            └─ 拼接向量文本
               → RagEmbeddingService.embed(text)
               → ChromaDB.addDocuments(...)
               → 写入 rag_vector_mapping

检索
    │
    ▼
Redis 缓存（key = rag:search: + MD5(query)）命中 → 直接返回
    │ 未命中
    ▼
RagEmbeddingService.embed(query)        ← 查询向量化
    │
    ▼
ChromaDB.queryByEmbedding(embedding, topK)
    │
    ▼（可选）
Reranking：调用 qwen-plus 对候选文档重排序
    │
    ▼
根据 metadata.docId 查 doc_issue 关联文档详情
    │
    ▼
写入 Redis 缓存（TTL = 1800s）→ 返回
```

### 2.2 分片策略

**当前实现：单分片模式（整篇文档 = 1 个向量）**

```typescript
// rag.service.ts — indexDoc 方法
const text = [doc.title, doc.problem, solutionSummary, doc.tags]
  .filter(Boolean)
  .join('\n');

const vectorId = `doc_${doc.id}_0`;   // chunk_index 固定为 0
```

| 分片要素 | 说明 |
|----------|------|
| 向量 ID 格式 | `doc_{docId}_{chunkIndex}`，如 `doc_42_0` |
| 分片序号 | 当前恒为 `0`，预留多分片扩展 |
| 向量文本内容 | `title + "\n" + problem + "\n" + solution前200字 + "\n" + tags` |
| solution 截断 | 超过 200 字截断后加 `"..."`，防止 Embedding 输入过长 |
| 数据库侧 | `rag_vector_mapping` 设计为一对多（doc_id → N 个 chunk），当前一对一 |

**为何保留 chunk_index 字段？**  
表结构预留了多分片能力（`chunk_index` 字段 + 一对多主键设计），方便后续对长文档做滑动窗口分片，无需改表结构。

### 2.3 向量存储（ChromaDB）

| 字段 | 说明 |
|------|------|
| ids | `doc_{docId}_0` 字符串 |
| embeddings | 通义千问 `text-embedding-v3` 生成的浮点数组（维度 1024 或 1536） |
| documents | 向量对应的原始文本（便于检索结果展示） |
| metadatas | `{ docId: number }` — 关联回 MySQL 文档 ID |

ChromaDB Collection 名称由 `CHROMA_COLLECTION` 环境变量配置，默认 `fe_knowledge`。

### 2.4 Embedding 模型

| 项目 | 说明 |
|------|------|
| 默认模型 | 通义千问 `text-embedding-v3` |
| 接口协议 | OpenAI 兼容格式（`openai` SDK 调用） |
| API 地址 | `https://dashscope.aliyuncs.com/compatible-mode/v1` |
| 支持方式 | 单条 `embed(text)` / 批量 `embedBatch(texts[])` |

### 2.5 Reranking 机制

```
候选文档（topK=20）
    │
    ▼
构造 Prompt：将每篇文档内容前 300 字拼为候选列表 [0][1][2]...
    │
    ▼
调用 qwen-plus（temperature=0）：
  系统提示：按相关性排序，返回 JSON 数组如 [2,0,1]
    │
    ▼
解析排序索引 → 重排结果 → 取 topN（默认5）返回
```

开关由 `sys_config.rag_rerank_enabled` 控制（`true/false`）。

### 2.6 Redis 缓存

| 项目 | 说明 |
|------|------|
| 缓存键 | `rag:search:` + MD5(query + topKOverride) |
| TTL | 默认 1800s，从 `sys_config.rag_cache_ttl` 读取 |
| 失效策略 | 仅 TTL 自然过期，文档更新/删除不主动清缓存 |
| 存储格式 | JSON 字符串序列化整个响应对象 |

## 三、数据存储设计

### 3.1 存储层分工

| 存储 | 职责 | 数据类型 |
|------|------|----------|
| MySQL 8.0 | 核心业务数据 | 结构化（用户、文档、分类、配置、钉钉消息） |
| ChromaDB | 向量检索 | 高维浮点向量 + 原始文本 + metadata |
| Redis 7 | 缓存加速 | RAG 搜索结果缓存（key-value JSON） |

### 3.2 数据流向

```
手动录入文档
    → MySQL doc_issue（status=0）
    → 审核通过（status=1）
    → doc.updated 事件
    → 向量化 → ChromaDB + MySQL rag_vector_mapping

钉钉群消息
    → MySQL dingtalk_message（handleStatus=0）
    → AI 提取
    → MySQL doc_issue（status=0，source=2）
    → 审核通过 → 向量化 → ChromaDB

用户检索
    → Redis 命中 → 直接返回
    → Redis 未命中
        → ChromaDB 向量检索
        → MySQL doc_issue 查文档详情
        → Redis 缓存写入
        → 返回
```

## 五、优化建议

### 5.1 RAG 检索质量优化（高优先级）

**① 长文档分片缺失**

当前单篇文档仅生成 1 个向量，`solution` 超过 200 字后直接截断，大量有效信息丢失。

建议改为滑动窗口分片：
```
分片策略：按段落分割，每片 500~800 字，相邻片重叠 100 字
chunk_index: 0, 1, 2... 分别入库 ChromaDB
每片 metadata 携带 { docId, chunkIndex, totalChunks }
```

`rag_vector_mapping` 表已支持一对多，无需改表，只需修改 `indexDoc` 逻辑。

**② 向量化前缺乏文本预处理**

Markdown 格式中含大量 `#`、`**`、`-` 等标记符，会干扰向量语义。

建议：向量化前先用正则去除 Markdown 语法符号，保留纯文本内容后再调用 Embedding API。

**③ 缓存失效策略不完善**

文档更新/删除后缓存不主动清除，TTL 内用户仍可搜索到已删除/已驳回的文档。

建议：`doc.updated` / `doc.deleted` 事件触发时，扫描并清除 `rag:search:*` 相关缓存键（可通过 Redis SCAN 命令匹配前缀清除）。

**④ Reranking 使用 LLM 成本高**

当前用 `qwen-plus` 做重排序，每次搜索多调用一次大模型，延迟增加约 1~3s，token 成本高。

建议：改用专用重排序模型，如 `bge-reranker-v2-m3`（可本地部署或调用专用 API），延迟更低、成本更低。

**⑤ Reranking 结果解析无容错告警**

```typescript
// 当前代码：解析失败静默回退
} catch {
  return results;  // 只 fallback，无任何告警
}
```

建议：解析失败时打印 WARN 日志，并上报监控（如 Hubble）。

### 5.2 数据库查询优化

**① 文档关键词搜索未用全文索引**

```typescript
// 当前实现（LIKE 全表扫描）
qb.andWhere('doc.title LIKE :keyword', { keyword: `%${query.keyword}%` });
```

`doc_issue` 表已建有 `FULLTEXT ft_title_problem(title, problem)` 全文索引，但代码中完全没用到。

建议改为：
```sql
WHERE MATCH(title, problem) AGAINST (:keyword IN BOOLEAN MODE)
```

可利用全文索引大幅提升搜索性能。

**② sys_config 每次都查数据库**

`RagService` 在每次检索时都分别查 3 次 `sys_config` 表（top_k、rerank_enabled、cache_ttl）。

建议：启动时加载到内存，或用 Redis 缓存（TTL = 5min），配置变更时主动失效。

**③ rag_vector_mapping 缺少复合索引**

查询 `WHERE doc_id = ? ORDER BY chunk_index` 场景无复合索引加速。

建议新增：`KEY idx_doc_chunk (doc_id, chunk_index)`

### 5.3 安全问题（需修复）

**① AppSecret 明文存储**

```sql
-- sys_dingtalk 表注释写"加密存储"，但代码直接明文存入
app_secret varchar(200) COMMENT '钉钉应用AppSecret（加密存储）'
```

建议：使用 AES 对称加密后存储，读取时解密，密钥由环境变量注入。

**② JWT Secret 有默认值**

```typescript
// config/index.ts
secret: env('JWT_SECRET', 'fe-knowledge-jwt-secret-key-2024'),
```

生产环境若未配置 `JWT_SECRET` 环境变量，将使用公开的默认密钥，存在安全风险。

建议：生产启动时校验 `JWT_SECRET` 是否已设置，未设置则拒绝启动。

**③ 登录接口缺乏独立限流**

当前全局 60s/100 次限流对登录接口过于宽松，建议对 `/api/auth/login` 单独设置：60s 内最多 5 次，防止暴力破解。

### 5.4 性能优化

**① 全量同步串行处理**

```typescript
// syncAll 当前实现
for (const doc of docs) {
  await this.indexDoc(doc);  // 串行，50篇文档需逐个等待 Embedding API
}
```

建议改为分批并发：
```typescript
const BATCH_SIZE = 5;
for (let i = 0; i < docs.length; i += BATCH_SIZE) {
  await Promise.all(docs.slice(i, i + BATCH_SIZE).map(d => this.indexDoc(d)));
}
```

**② Redis 连接未复用全局连接池**

`RagService` 在构造函数中 `new Redis(...)` 创建独立连接，未使用全局连接池。

建议：将 Redis 客户端提取为全局 Provider，通过 DI 注入，避免多个 Service 各自创建连接。

**③ syncAll 没有进度反馈**

全量同步文档数量多时前端无法感知进度，易超时。

建议：引入 Server-Sent Events（SSE）或 WebSocket 推送同步进度，或改为后台异步任务 + 轮询状态接口。

---

### 5.5 钉钉采集优化

**① batchScan 一次性加载全部未处理消息**

```typescript
const unprocessed = await this.messageRepo.find({ where: { handleStatus: 0 } });
```

消息量大时可能导致内存压力。建议改为分批游标查询，每批处理 50 条。

**② createDocFromResult 硬编码 createUserId=1**

```typescript
createUserId: 1,  // 硬编码系统用户
```

建议：自动采集来源的文档统一用系统账号（ID=1）是可接受的，但应提取为命名常量 `SYSTEM_USER_ID = 1`，避免魔法数字。

**③ 钉钉 AccessToken 仅内存缓存**

进程重启后缓存失效，重启时首次收到群消息可能因 Token 获取延迟而处理缓慢。

建议：将 AccessToken 缓存写入 Redis，进程重启后仍可复用未过期 Token。

**④ 群 ID 过滤逻辑**

```typescript
if (data.conversationId && allowed.length > 0 && !allowed.includes(data.conversationId))
```

当 `groupIds` 为空时不过滤（接收所有群消息），这是预期行为，但建议在配置界面给出明确提示。

### 5.6 功能完善建议

| 缺失功能 | 建议 |
|----------|------|
| 文档全文检索 | 利用已建的 ngram 全文索引实现，同时支持关键词高亮 |
| 图片/文件上传 | 配置类中已有 `upload.dir` 和 `maxFileSize`，但功能未实现；Markdown 编辑器需配合图片上传接口 |
| 操作审计日志 | 关键操作（审核、删除、权限变更）应写入操作日志表，供管理员追溯 |
| 文档审核通知 | 文档审核通过/驳回后，通过钉钉机器人通知文档创建人 |
| 统计增强 | 首页仪表盘仅有4个指标，建议增加：分类文档分布、近30天新增趋势折线图、热门搜索词 |
| 向量库健康检查 | 健康检查接口只检查 MySQL + 磁盘，未检测 ChromaDB 可用性，建议补充 |
| Token 黑名单 | 用户退出登录后 accessToken 在过期前仍有效，建议退出时将 Token 加入 Redis 黑名单 |
| 文档版本历史 | 文档编辑无版本记录，无法回滚历史版本 |
