# 前端踩坑知识库 RAG 系统 - 完整技术方案

> 本文档基于实际代码梳理，可直接对照代码落地开发，不含未实现的空谈内容。

---

## 一、系统定位与目标

**定位**：轻量级前端踩坑知识库，融合文档管理 + RAG 智能检索 + 钉钉群消息自动采集。

**核心目标**：
1. 文档分类、增删改查、模糊搜索、审核发布
2. RAG 向量检索，输入问题返回相似问题 + 解决方案
3. 钉钉群消息自动采集，AI 提取后生成待审核文档

**核心用户**：前端开发团队成员、技术负责人

## 二、系统架构

### 1. 分层架构

```
┌───────────────────────── 前端层 ─────────────────────────┐
│  Vue 3 + Vite + Element Plus + md-editor-v3              │
│  路由守卫(角色)、Pinia 状态、Axios Token 自动刷新         │
└──────────────────────────────────────────────────────────┘
                          ↕ HTTP / JWT Bearer
┌───────────────────────── 网关层 ─────────────────────────┐
│  NestJS：CORS、全局 JWT Guard、角色 Guard、限流 Guard     │
│  ValidationPipe、HttpExceptionFilter、TransformInterceptor│
└──────────────────────────────────────────────────────────┘
                          ↓
┌───────────────────────── 服务层 ─────────────────────────┐
│  AuthModule │ UserModule │ CategoryModule │ DocModule     │
│  RagModule  │ DingtalkModule │ HealthModule               │
│  EventEmitterModule（事件总线解耦文档生命周期）            │
└──────────────────────────────────────────────────────────┘
                          ↓
┌───────────────────────── 数据层 ─────────────────────────┐
│  MySQL 8.0   — 结构化业务数据（TypeORM）                  │
│  Redis 7     — 缓存 / 限流 / Token 黑名单                │
│  ChromaDB    — 向量检索                                   │
│  本地/MinIO   — 文件存储                                   │
└──────────────────────────────────────────────────────────┘
```

### 3. 全局中间件链

请求经过以下全局处理链（在 `app.module.ts` 中注册）：

```
请求 → ValidationPipe(参数校验)
     → JwtAuthGuard(认证，@Public跳过)
     → RolesGuard(角色授权，@Roles约束)
     → ThrottlerGuard(限流: 60秒100次)
     → Controller → Service
     → TransformInterceptor(包装为 ApiResponse)
     → HttpExceptionFilter(异常统一处理)
```

### 4. 统一响应格式

所有接口返回格式：

```json
{
  "code": 0,
  "message": "操作成功",
  "data": {},
  "timestamp": 1716285600000
}
```

分页响应 data 结构：

```json
{
  "list": [],
  "total": 100,
  "page": 1,
  "pageSize": 20
}
```

## 四、接口设计（完整）

所有接口前缀 `/api`，除标注 `@Public` 外均需 JWT Bearer Token。

### 4.1 认证接口 `/api/auth`

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| POST | /auth/login | 登录，返回 accessToken + refreshToken | 公开 |
| POST | /auth/refresh | 刷新 Token | 公开 |
| GET | /auth/info | 获取当前用户信息 | 登录 |
| PUT | /auth/password | 修改密码 | 登录 |

**登录请求**：
```json
{ "username": "admin", "password": "admin123" }
```

**登录响应**：
```json
{
  "code": 0,
  "data": {
    "accessToken": "eyJ...",
    "refreshToken": "eyJ..."
  }
}
```

**Token 机制**：
- accessToken 有效期 15 分钟
- refreshToken 有效期 7 天
- refreshToken 中包含 `type: 'refresh'` 标识，不可用于访问接口
- 密码修改要求：最少 8 位，必须包含字母 + 数字

## 五、RAG 向量检索方案

### 1. 架构流程

```
┌─────────────────── 文档入库流程 ───────────────────┐
│                                                     │
│  文档发布 → EventEmitter('doc.created')              │
│         → RagService.onDocCreated()                 │
│         → 拼接文本: title + problem + solution摘要   │
│         → EmbeddingService.embed(text)              │
│         → ChromaService.addDocuments()              │
│         → 保存 rag_vector_mapping 映射              │
│                                                     │
└─────────────────────────────────────────────────────┘

┌─────────────────── 检索流程 ───────────────────────┐
│                                                     │
│  用户输入问题                                        │
│  → MD5(query) 查 Redis 缓存                         │
│  → 命中：直接返回缓存结果                            │
│  → 未命中：                                         │
│    → EmbeddingService.embed(query)                  │
│    → ChromaService.queryByEmbedding(embedding, topK)│
│    → [可选] Reranking: 大模型重排序                  │
│    → 查询关联的 doc_issue 文档详情                    │
│    → 写入 Redis 缓存 (TTL=3600s)                    │
│    → 返回结果                                       │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 2. 向量化内容

```
向量内容 = 文档标题 + 问题描述 + 解决方案摘要(前200字) + 标签
```

solution 是核心价值，必须参与向量化。超过 200 字截断，避免 embedding 输入过长。

### 3. Embedding 模型

- **推荐**：通义千问 `text-embedding-v3`
- **接口**：OpenAI 兼容格式，统一使用 `openai` SDK 调用
- **备选**：开源 `bge-large-zh-v1.5`（可本地部署）
- **调用方式**：`RagEmbeddingService` 封装，支持单条 / 批量

### 4. Reranking（可选开关）

检索 top_k（默认 20）结果后，调用大模型做重排序：
- 将候选文档内容发送给 AI，让其按相关性排序
- AI 返回排序后的索引数组 `[2,0,1,...]`
- 取 top_n（默认 5）返回

通过 `sys_config` 表的 `rag.rerank_enabled` 键控制开关。

### 5. 缓存策略

- 缓存 key：`rag:search:` + MD5(query)
- 缓存 TTL：3600 秒（可配置）
- 存储：Redis
- 文档更新/删除时不主动清缓存（TTL 自然过期，简化逻辑）

### 6. 数据同步策略

| 文档事件 | 操作 |
|----------|------|
| 创建（status=1 已发布） | 向量化入库 + 保存映射 |
| 更新 | 删除旧向量 → 重新入库 |
| 删除 | 删除向量 + 删除映射 |
| 审核通过 | 同 "创建"（通过事件触发） |

## 六、钉钉自动采集方案

### 1. 技术方案

使用**钉钉企业内部应用 + Stream API**（非 webhook），原因：
- webhook 只能**发送**消息到群，不能**接收**群消息
- Stream API 基于长连接，无需公网域名，开发调试方便

### 2. 配置步骤

1. 登录 [钉钉开放平台](https://open.dingtalk.com)
2. 创建企业内部应用，获取 AppKey / AppSecret
3. 开启机器人功能，获取 RobotCode
4. 在系统中填入配置并启用

### 3. 消息处理流程

```
钉钉群消息
  → DWClient(Stream) 接收
  → 解析 conversationId、发送人、内容
  → 检查是否为监听群（groupIds 过滤）
  → 存入 dingtalk_message 表（handleStatus=0）
  → 管理员在后台查看消息列表
  → 手动点击"生成文档"
    → 调用大模型提取 title/problem/solution/tags
    → 创建 doc_issue（status=0 待审核, source=2）
    → 更新消息 handleStatus=1, 关联 docId
  → 管理员审核通过 → 文档发布 → 自动向量化
```

### 4. AI 提取提示词

```
你是一个前端技术文档整理专家，请把下面的聊天消息整理成标准的前端问题文档：
1. 问题标题（简洁精准）
2. 问题描述（清晰描述问题现象）
3. 解决方案（详细、可直接使用）
4. 技术标签（最多3个）
输出格式严格使用JSON，不要其他内容：
{"title":"","problem":"","solution":"","tags":""}
```

## 七、安全设计

### 1. 认证机制

| 项目 | 说明 |
|------|------|
| 认证方式 | JWT Bearer Token |
| accessToken | 有效期 15 分钟 |
| refreshToken | 有效期 7 天，payload 含 `type: 'refresh'` |
| Token 刷新 | 前端 Axios 拦截器自动处理 401，队列化并发请求 |
| 密码加密 | bcrypt（10 轮 salt） |
| 密码策略 | 最少 8 位，必须包含字母 + 数字 |

### 2. 权限控制

| 维度 | 实现方式 |
|------|----------|
| 接口权限 | `@Roles('admin')` 装饰器 + RolesGuard |
| 路由权限 | `meta.roles` + 路由守卫（前端） |
| 菜单权限 | 侧边栏根据 `userStore.isAdmin` 动态显示 |
| 公开接口 | `@Public()` 装饰器跳过 JWT 校验 |

### 3. 安全防护

| 防护项 | 实现方式 |
|--------|----------|
| XSS | Markdown 渲染使用 `marked` + `DOMPurify.sanitize()` |
| 接口限流 | `@nestjs/throttler`，60 秒内最多 100 次 |
| 参数校验 | `ValidationPipe` + `class-validator`（白名单模式） |
| 敏感配置 | 环境变量注入（`.env`），禁止硬编码 |
| SQL 注入 | TypeORM 参数化查询 |

| `useCategoryStore` | 分类树缓存 |
| `useAppStore` | 侧边栏折叠状态、全局 loading |

### 3. 请求封装

Axios 拦截器实现 Token 自动刷新：
- 401 响应 → 用 refreshToken 换新 accessToken
- 刷新期间并发请求入队等待
- 刷新成功 → 更新 Token + 重放队列
- 刷新失败 → 清除 Token + 跳转登录

## 九、事件驱动架构

使用 NestJS `@nestjs/event-emitter` 模块解耦文档生命周期：

| 事件 | 触发点 | 监听方 |
|------|--------|--------|
| `doc.created` | DocService.create() / DingtalkService.generateDoc() | RagService.onDocCreated() |
| `doc.updated` | DocService.update() | RagService.onDocUpdated() |
| `doc.deleted` | DocService.remove() / batchDelete() | RagService.onDocDeleted() |

好处：文档模块不感知 RAG 模块的存在，未来增加通知、日志等能力只需新增监听器。

## 十、部署方案

### 1. Docker Compose 一键部署

```yaml
服务:
  mysql    → 3306  数据库（自动执行 init.sql）
  redis    → 6379  缓存
  chroma   → 8000  向量库
  server   → 3000  NestJS 后端
  web      → 80    Nginx + Vue3 前端
```

### 2. 启动命令

```bash
cp .env.example .env
# 编辑 .env 填入 AI_API_KEY 等配置
docker compose up -d
```

### 3. Nginx 配置要点

- SPA 路由回退：`try_files $uri $uri/ /index.html`
- API 反向代理：`/api/` → `http://server:3000`
- 静态资源缓存：`/assets/` 30 天强缓存

### 4. 健康检查

- 后端：`GET /api/health`（检查 MySQL 连接 + 磁盘空间）
- Docker healthcheck 配置在 `docker-compose.yml` 中

### 5. 运行环境要求

- 服务器：2 核 4G（50 人以内团队够用）
- 系统：Ubuntu 20.04+ / CentOS 7+
- Docker 20.10+, Docker Compose 2.0+
